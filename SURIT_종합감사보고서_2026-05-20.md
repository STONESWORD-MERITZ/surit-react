## 3. 분석 정확도 교차검증 — 실제 PDF 실측

서은희(어머니) 진료 PDF 5건(기본 2·세부 2·처방조제 1, 총 1,368 레코드)을 현행 `a598e98` 코드의 결정론 경로(파싱 → 질병집계 → 필터 → 메리츠)에 직접 통과시켜 검증했습니다. AI(Gemini)는 호출하지 않고 코드 로직만 실측했습니다.

### 3.1 [P0-1] `_max_presc` 투약일수 — SUM/MAX 불일치 실측 확인

`_max_presc` 함수는 같은 시그니처로 **두 곳에 별도 구현**되어 있고, 동작이 다릅니다.

- `pipeline/helpers.py:425` — 항상 `max(values)` 반환 (정상, "수정 완료"분)
- `filters.py:74` — 에피소드가 중첩 dict면 `sum(values)` 반환 (**미수정 — 버그 잔존**)

```python
# filters.py:87-89  (버그 잔존)
if any(isinstance(v, dict) for v in med_dict.values()):
    return sum(values)        # ← 모든 처방 에피소드를 합산
return max(values)
```

실제 분석 경로(`filters._build_health` / `_build_easy`)는 항상 중첩 dict(`med_dates_pharma_episode`)를 입력으로 받으므로 **운영 시 언제나 합산 분기**를 탑니다. 서은희 데이터 실측 발산 사례:

| 질병 | filters(SUM) | helpers(MAX) | 원본 에피소드 |
|---|---|---|---|
| K297 상세불명의 위염 | **10** | 5 | 2025-12-10 밀양병원 2일 / 2022-03-02 유가제일내과 3일·현풍경대비뇨기 5일 |
| N760 급성질염 | **11** | 6 | 2025-07-21 5일 / 2021-04-08 6일 |
| H359 상세불명의 망막장애 | **122** | 58 | 다수 안약 처방 합산 |
| N300 급성방광염 | **23** | 8 | 다수 |
| R093 이상가래 | **8** | 4 | — |
| U071 코로나19 | **10** | 7 | — |

**영향** — ① 결과 카드의 투약일수 표시(`result_builder`는 `helpers` = MAX 사용)와 필터 룰이 본 값(`filters` = SUM)이 **같은 질병에서 서로 다릅니다.** 설계사는 한 화면에서 "투약 122일", 다른 곳에서 "58일"을 보게 됩니다. ② 필터 룰 `R-H-Q3-MED-30D`(30일 이상 투약)이 **합산값으로 발화**하므로, 6일짜리 처방을 6번 받은 환자(MAX 6일)가 SUM 36일로 잡혀 거짓 양성이 발생합니다. 서은희 데이터에서는 30일 임계를 뒤집는 사례가 우연히 없었으나, 표시 수치는 이미 부정확하며 다른 환자에서는 발화 자체가 뒤집힙니다.

**조치** — `filters.py`의 인라인 중복 6종(`_max_presc` 포함)을 삭제하고 `helpers`에서 import. "순환 import 회피"라는 코드 주석은 사실과 다릅니다(`helpers`는 `filters`를 import하지 않음). 회귀 테스트 `test_med_days_takes_max_not_sum`도 `helpers`판만 검증 중이라 `filters`판 버그를 잡지 못합니다 — `filters._max_presc`도 테스트에 추가.

### 3.2 [P0-3] 건강체 Q3 조회기간 = 10년 (표준 청약서는 5년)

실측 결과 건강체 결과 7건이 전부 "10년 이내 …"로 산출됐고, 필터 룰 ID도 `R-H-Q3-INP-10Y`/`-SURG-10Y`/`-VISIT-7`/`-MED-30D`, 결과 라벨도 `result_builder.py:101`에서 `"[3번질문] 10년 이내 입원/수술/7회이상통원/30일이상투약"`로 확인됐습니다. 간편심사 Q2(입원/수술)도 동일하게 10년입니다(`result_builder.py:93`).

표준화된 손해보험·생명보험 청약서의 알릴의무 3번 문항은 통상 **"최근 5년 이내" 입원·수술·계속하여 7일 이상 치료·계속하여 30일 이상 투약**입니다. 별도의 "10년" 문항은 표준 청약서에 존재하지 않습니다.

**결정 필요** — 두 가지 중 하나로 정리해야 합니다.
- (A) 표준에 맞춰 Q3를 5년으로 수정. 가장 명확하며 청약서 문항과 1:1 대응.
- (B) 10년 조회를 의도적 보수 설계로 유지하되, `[3번질문]` 같은 청약서 문항번호 라벨을 떼고 "참고용 10년 이력 조회"로 분리 표기. 현재처럼 청약서 문항번호를 붙인 채 10년을 쓰면, 설계사가 8년 전 이력을 실제 Q3 답변에 그대로 옮겨 과잉 고지로 이어집니다.

권고: (A). 출시 전 결정.

### 3.4 점검 권고 — 메리츠 평가 결과의 타당성

서은희(질병 175그룹, 입원·수술 이력 포함) 실측에서 메리츠 평가가 `eligible=True / 예외질환 0건 / recommended_year=None`으로 나왔습니다. 예외질환표(118룰)에 매칭되는 질병이 없으면 0건이 정상일 수 있으나, 입원·수술 이력이 있는 다질병 케이스가 곧바로 "eligible=True"로 나오는 것이 의도된 결과인지 확인이 필요합니다. 또한 준법감시 감사에서 룰 테이블 전 항목의 경과기간이 일괄 `7일`로 고정돼 있어(실제 약관은 질환군별로 상이) 출처·정확성 검증이 선행돼야 합니다.

- **건강검진·예방접종(Z·U코드)이 결정론 경로에서 제외되지 않음** — 제외 로직이 AI 프롬프트 지시로만 존재하고, 코드 경로(`build_disease_stats`/`build_code_based_items`)에는 없습니다. `Z00`(일반검진)·`Z11`(선별검사)·`Z23~Z27`(예방접종)이 KCD 패턴을 통과해 질병그룹이 되고, 3개월 내 방문 시 Q1을 발화시킬 수 있습니다. 비질병 KCD 접두 차단 목록을 `_is_valid_disease`에 추가 필요.
- **`_pharma_seen` 중복 제거 키가 `(날짜, 약품명)`뿐** — 같은 날 같은 약을 다른 병원/약국에서 받은 정상 처방이 누락될 수 있고, seen-key가 데이터가 기록되는 그룹과 다른 스코프에 기록돼 이후 정상 행을 잘못 스킵할 수 있음. 키에 병원명(및 투약일수) 포함 필요.
- **빈 PDF/이미지 PDF 오류 메시지 혼동** — 정상 복호화됐지만 표가 없는 PDF가 "비밀번호 확인" 안내로 떨어짐.
- **처방 PDF 오분류 가능** — 헤더 키워드가 OCR로 누락되면 처방조제 파일이 `detail`로 분류될 수 있음. 페이지 텍스트 신호가 `pharma`인데 헤더가 `detail`이면 페이지 텍스트를 신뢰하도록 보정 필요.
- **창 경계 오프바이원** — `_dts_in_range`는 기준일 당일 포함(`>=`), `analyzer.py` 태깅은 제외(`>`). 두 경로가 경계일에서 하루 어긋남. 한 가지 규칙으로 통일.
- **`AnalysisProgress` 컴포넌트가 만들어졌으나 미사용** — 분석 중 화면이 최대 3분간 빈 상태(아래 UX P0와 동일 사안).

### P2 (선별)
- 5년/10년 컷오프가 고정 1825/3650일이라 윤년 2~3일 누락.
- 진료시작일이 비어 있는 행이 경고 없이 조용히 누락.
- 초대용량 PDF에서 `filtered_lines[:800]`·`30,000자` 잘림이 사용자 경고 없이 발생.

### 누락된 테스트 (추가 권장)
`filters._max_presc` 직접 검증 / 창 경계 정확일(90·365·1825·3650일 전) / 미래일자 행이 실제로 드롭되는지 / Z코드 제외 / 같은 날 같은 약 다른 병원 / `compute_prescription_end_dates` / `detect_drug_changes` / `parse_nhis_text` / `result_builder` 병합 로직 / 메리츠 `_compute_recommended_disclosure` 추천연도 계산.

## 6. 보안 감사 — API·인증·의존성·PII

### P0
- **`/api/analyze` 완전 무인증** — 토큰·API키·세션 검사 전무. 프런트 `ProtectedRoute`는 React 라우트만 막을 뿐 별도 배포된 Railway 백엔드에는 효력이 없습니다. Railway URL은 `.env.example`·`ci.yml`·CSP에 공개돼 있어, 누구나 직접 PDF를 POST해 Gemini 호출 비용을 발생시킬 수 있습니다(denial-of-wallet). FastAPI 의존성으로 Supabase JWT를 서버에서 검증하고, 프런트가 `Bearer` 토큰을 첨부하도록 수정.
- **업로드 크기·개수 제한 없음** — `await f.read()`로 파일 전체를 메모리에 적재. 무인증과 결합 시 대용량 PDF 다수 업로드로 Railway 컨테이너를 OOM 종료시킬 수 있음. 파일당/총합 크기·개수 상한을 두고 413으로 거절.

### P1 (주요)
- **CORS 기본값이 localhost 포함** — `CORS_ORIGINS` 미설정 시 운영에서 localhost 출처의 credentialed 요청을 허용. 운영 환경에서는 localhost를 빼고, 미설정 시 기동 실패하도록.
- **시크릿** — `.env`에 실제 Gemini API 키가 평문 보관(단, git 커밋·번들 유출은 없음 — `.gitignore` 정상). GCP에서 키 사용 API를 Generative Language API로 제한하고 Railway 환경변수로만 관리 권장. (Supabase anon 키가 번들에 포함되는 것은 정상 — 공개 키.)
- **레이트리밋 우회 가능** — slowapi가 `get_remote_address` 사용. Railway 프록시 뒤에서 전역 단일 버킷이 되거나 `X-Forwarded-For` 스푸핑으로 우회됨. 신뢰 프록시 인지 키 함수 사용, 인증 도입 후 사용자 ID 기준으로 변경.
- **백엔드 의존성 전면 미고정** — `requirements.txt`가 `fastapi`·`uvicorn`·`pdfplumber`·`pandas`·`google-genai`를 버전 무고정. 배포 시점마다 다른 버전 — 재현 불가·롤백 불안정. 정확 버전 고정 + 락파일.
- **회원가입 비밀번호 6자** — 의료 민감정보 도구 계정으로는 약함. 10자 이상 + Supabase 유출 비밀번호 차단 활성화.
- **Supabase RLS 미검증** — 프런트가 anon 키로 DB에 직접 접근하므로 모든 테이블의 RLS가 유일한 보호막. 대시보드에서 전 테이블 RLS 활성화·`auth.uid()` 스코프 확인 필요(코드만으로 확인 불가).
- **파일 시그니처 미검증** — `%PDF-` 매직바이트 확인 없이 바로 파싱.

### P2 (선별)
- `/api/health`가 env·커밋 SHA 등 내부정보 노출.
- Sentry 로컬변수 캡처로 스택프레임 안에 질병 데이터가 들어갈 수 있음 — `include_local_variables=False` 권장.
- 잡파일이 git 추적 중 — `backend/=1.0.0`(pip 로그가 잘못 커밋됨), `backend/analyzer_backup.py`(데드 백업).

### 양호
시크릿 git 미커밋, `.gitignore` 정상, Sentry PII 스크러빙(프런트·백엔드)·세션 리플레이 비활성화, 업로드 PDF 메모리 처리 후 즉시 해제, `vercel.json`의 CSP·HSTS·X-Frame-Options 설정.

## 8. 운영 준비도 감사

### P0
- **Gemini 429(쿼터 소진) 미재시도** — 재시도 조건이 `503`/`UNAVAILABLE`/`high demand`만. 429(`RESOURCE_EXHAUSTED`)는 즉시 치명 처리. 무료 티어는 분당 호출 한계가 낮고 앱은 한 번에 최대 6개 동시 호출 — 출시일 트래픽 급증 시 가장 먼저 터질 지점. 재시도 대상에 `429`/`RESOURCE_EXHAUSTED`/`quota` 추가 + Gemini 유료 티어 확보.
- **업로드 크기 제한 없음** — 보안 P0과 동일.

### P1 (주요)
- **CI가 배포를 막지 못함** — Vercel·Railway가 git push마다 독립 자동배포. CI 실패와 무관하게 배포될 수 있음. GitHub 브랜치 보호로 CI 통과를 머지 필수조건으로.
- **백엔드 의존성 미고정** — CI 통과 ≠ 운영 동작. 정확 버전 고정.
- **Supabase 환경변수 미설정 시 앱 전체 다운** — `supabase.ts`가 가드 없이 `createClient` 호출. 시작 시 검증 추가.
- **`/api/health`가 얕은 검사** — Gemini·Supabase 도달성을 확인하지 않고 키 문자열 존재만 확인. 깊은 헬스체크 추가.
- **Sentry `release` 미설정** — 어느 배포에서 난 오류인지 추적 불가. `RAILWAY_GIT_COMMIT_SHA`/Vercel 커밋 SHA로 release 태깅.
- **서버측 요청 타임아웃이 클라이언트와 불일치** — 프런트는 180초에 중단하지만 백엔드는 최대 ~20분 작업 지속 가능. 워커·쿼터 낭비. `asyncio.wait_for(~170초)`로 상한.
- **콜드 스타트** — Railway 슬립 티어면 첫 사용자가 20~60초 대기. 5초 워밍업 fetch는 거의 타임아웃. 유료 티어 이전 또는 외부 핑.

### P2 (선별)
- Gemini 실제 요청 경로(`ai_judgment.py`)에 자동 테스트 0 — 429/503/타임아웃/깨진 JSON 시뮬레이션 테스트 추가.
- 구조화 로깅·상관 ID 부재 — Sentry 이벤트와 서버 로그를 못 잇음.
- 스테이징 환경·배포 후 스모크 테스트 부재.

## 12. 부록 A — 감사 환경 이슈 (코드 결함 아님)

감사 중 확인된 작업환경 문제로, SURIT 코드 자체 결함과 구분해 정리합니다.

1. **파일 손상 (복구 완료)** — 감사 착수 시 작업 트리의 다수 파일이 이전 세션의 편집 도중 잘려 있어 백엔드가 기동 불가 상태였습니다. 전체 작업 트리를 커밋 `a598e98` 기준으로 복원했고, 현재 작업 트리 내용은 HEAD와 100% 일치(불일치 0건), pytest 68건 통과로 정상화했습니다.

2. **git 인덱스 손상** — `git status`가 38개 파일·3,420줄 삭제처럼 보이고 `git ls-files`가 `./`만 반복 출력합니다. 이는 **인덱스 손상**이며 실제 파일 변경이 아닙니다(작업 트리 내용은 HEAD와 동일함을 직접 검증). 운영 준비도 감사가 "미리뷰되지 않은 리팩터가 대기 중"으로 본 것은 이 인덱스 손상에 따른 오탐입니다. 복구 방법 — PowerShell에서:
   ```
   Remove-Item .git\index -Force
   git reset
   ```

3. **정리 필요 파일** — 감사 중 복원 과정에서 `backend/keywords.json.restored`(2,499바이트, 유효 JSON 사본) 한 개가 남았습니다. 무해하나 불필요하므로 삭제 권장. 함께 정리할 잡파일: `backend/=1.0.0`(pip 로그 오커밋), `backend/analyzer_backup.py`(데드코드), 루트의 빈 `새 텍스트 문서.txt`.
   ```
   Remove-Item backend\keywords.json.restored, backend\=1.0.0 -Force
   ```
